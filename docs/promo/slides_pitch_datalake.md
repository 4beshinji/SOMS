# 都市データレイク構想

**プライバシーコスト 0 の都市知能基盤**

秘密計算 × Edge AI × 分散 Federation
個人データを一切収集せずに、都市規模のデータ活用を実現する。

SOMS — Symbiotic Office Management System

# 都市データの構造的ジレンマ

都市のCO2、人流、エネルギー消費を分析したい。
しかし、**欲しいのは集約統計なのに、取得を強いられるのは個人データ**だ。

| 欲しいもの | 取得を強いられるもの |
|---|---|
| 「オフィス街の平均CO2: 620ppm」 | 特定ビルの在室パターン |
| 「住宅街→商業地区の人流」 | 個人の移動軌跡 |
| 「地域のエネルギー消費波形」 | 各テナントの稼働状況 |
| 「都市全体の換気効率指標」 | 従業員の出退勤パターン |

1万拠点 × 全従業員から同意を取得するコスト: **実質的に無限大**

# 発想の転換

> 同意を「取りやすくする」のではなく、**「不要にする」**。

```
データ収集 → 同意取得 → 利用 → 管理義務
```

**個人データを「匿名化する」のではない。個人データが「物理的に存在しない」。**

# Layer 2: Local Differential Privacy

### ESP32で動く秘密計算

```python
# edge/lib/local_dp.py (MicroPython)
import urandom, math

def laplace_noise(sensitivity, epsilon):
    u = (urandom.getrandbits(32) / 0xFFFFFFFF) - 0.5
    scale = sensitivity / epsilon
    return -scale * math.copysign(1, u) * math.log(1 - 2 * abs(u))

def privatize(value, sensitivity, epsilon):
    return value + laplace_noise(sensitivity, epsilon)
```

**計算コスト**: 乱数生成 + 対数 + 乗算。ESP32 (240MHz) で 0.1ms。
**必要なもの**: `urandom` と `math`。暗号学の博士号は不要。

| チャネル | 感度 (sensitivity) | ε | ノイズ幅 | 意味 |
|---|---|---|---|---|
| temperature | 5.0 ℃ | 1.0 | ±5.0 ℃ | 個人の体温影響を隠蔽 |

### 集約による精度回復

- 1拠点: ±200ppm のノイズ
- 10拠点平均: ±63ppm (√10 で縮小)
- 100拠点平均: ±20ppm (√100 で縮小)

**拠点が増えるほど、プライバシーを維持しつつ精度が向上する。**

# Layer 3: Secure Aggregation

Region Hub 間の値を互いに非開示のまま合算する。

```
Hub A: x_A + mask_A ──┐
Hub B: x_B + mask_B ──┼→ 合計: x_A + x_B + x_C
Hub C: x_C + mask_C ──┘  (mask は打ち消し合い = 0)
```

### Pairwise Masking

1. 各ペアが共有秘密を持つ (DH鍵交換)
2. 共有秘密からマスク値を生成 (PRG)
3. 一方が +mask、他方が -mask を使用
4. 全マスクの合計は常に 0

**City Aggregator は合計値のみ取得。個別 Hub の値は数学的に復元不可能。**

---

# ε（イプシロン）: プライバシーの通貨

| ε | 保護強度 | 攻撃者の優位性 | 用途 |
|---|---|---|---|
| 0.1 | 極めて強い | e^0.1 ≒ 1.10倍 | 在室情報等の高感度データ |
| 0.5 | 強い | e^0.5 ≒ 1.65倍 | 標準的なセンサーデータ |
| 1.0 | 十分 | e^1.0 ≒ 2.72倍 | 環境データ (温度, CO2) |
| 2.0 | 限定的 | e^2.0 ≒ 7.39倍 | 公共性の高い集計のみ |

### Privacy Budget

- 各拠点 × 各チャネルに **ε予算** を割り当て
- クエリごとにεを消費。**使い切ったらデータ送出を停止**
- 時間窓（1時間）でリセット（ストリーミングデータの特性を活用）
- 消費状況は **公開監査ログ** で検証可能

# 法的フレームワーク: 技術が法を満たす

| | 生データ (Layer 1) | DP処理済み (Layer 2+) |
|---|---|---|
| **データ例** | 「3F会議室に5人在室」 | 「ゾーンA: 4.7 ± 1.2人」 |
| **個人情報保護法** | 該当 (特定個人を識別可能) | **非該当** (識別不可能) |
| **GDPR** | Personal Data | **Anonymous Information** (Recital 26) |
| **同意** | 必要 | **不要** |
| **越境移転制限** | あり | **なし** |

### 法的根拠

- **個情法 2条1項**: 「特定の個人を識別することができない」→ DP が数学的に保証
- **GDPR Recital 26**: 匿名化データは GDPR 適用範囲外
- **GDPR Art. 25**: Data Protection by Design → Edge DP が完全に該当

# 同意の構造的解消

```
従来:
  数千拠点 × 数万人 → 全員から個別同意 → 実務上不可能

SOMS:
  Site 内: 雇用契約・利用規約で処理 (既存の枠組み)
       ↓
  Site → Region: DP処理済み → 個人情報に非該当 → 同意不要
       ↓
  Region → City: Secure Aggregation → 同意不要
```

| 層 | 同意の範囲 | コスト |
|---|---|---|
| Layer 1 (Site) | 施設利用者 (既存の雇用契約) | **既存** |
| Layer 2 (Region) | 拠点間の統計共有 (業務契約) | **低** |
| Layer 3 (City) | **同意不要** | **0** |

都市レベルの同意取得コスト: **$0**
